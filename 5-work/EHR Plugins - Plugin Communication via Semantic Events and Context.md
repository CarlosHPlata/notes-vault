# ðŸ“ƒ Plugin Communication via Semantic Events and Context 

---
CDA UI Plugins operate in a sandboxed, micro frontend environment. Because they donâ€™t follow standard React prop/state flows, communication between plugins and between the host shell, is handled through a **semantic event system** and a **Plugin Context** abstraction.

## Plugin Context
Every plugin receives a `Plugin Context` object on initialization, this context is **inmutable** and contains:
- Initial metadata (ex. patient ID).
- Predefined helpers to handle event listening and dispatching.

> [!danger] Context is inmutable
> Plugins must not rely on updates to context values, **state changes are communicated through events** 

## Custom Events
Plugins dispatch and listen to **custom semantic events** to interact with the environment.

> [!warn] Key Rules
> - Events should be prefixed with the plugin's name for traceability.
> - Use `PluginCustomEvent.CUSTOM` from the context to dispatch.
### Example:
```ts
context.dispatchEvent(context.PluginCustomEvent.CUSTOM, {
  eventName: EventIds.PATIENT_CHART_OPEN_DRAWER,
  auxiliaryProps: {
    pluginName: 'myplugin',
    preferredWidth: '35%',
    pluginParameters: {
      data: { viewMode: 'ADD_PROBLEM' }
    }
  }
});
```

## Use Case: Property Change Simulation
Since plugins **cannot receive live prop updates**, one common pattern is:
- Initialize with `PluginContext`.
- Initialize with Query Param.
- Fetch the data.
- **React to external changes** from the custom events, or built-in events.
