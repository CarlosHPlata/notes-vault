**(A*)** is an **informed search** algorithm that finds the **shortest path** to a goal in a [[Data Structures - Graphs|Graph]]. By defining a function $g(n)$ that returns the **actual cost so far** and an estimated **cost to goal** by an [[Algorithms - Heuristics in search algorithms|heuristic function]] $h(n)$. It balances exploration like [[Algorithms - BFS|BFS]], goal direction like [[Algorithms - Greedy Best First Search|Greedy Best-First]] and the estimated cost $g(n)$ from [[Algorithms - Dijkstraâ€™s]] using the formula:
$$f(n) = g(n) + h(n)$$
- $g(n)$: The cost from the **start node** to the current node.
- $h(n)$: Heuristic estimate from the current node to **the goal**.
- $f(n)$: Total estimated cost of a path through $n$.

> [!tip] Search Algorithms
> A* is part of algorithms called **Search Algorithms** designed to traverse and search values in [[Data Structures - Graphs|Graphs]] more specific is part of the so called [[Ai - Informed vs Uninformed Search problems|Informed Search Algorithms]].
## Best of all
You might assume, after reading [[Algorithms - Greedy Best First Search]], that it always finds the shortest path; but that would be a mistake. While Greedy Best-First Search **feels efficient** by choosing the node that seems closest to the goal (based on $h(n)$), it **doesnâ€™t account for the cost of getting there**.

Letâ€™s examine the following case:

![[Pasted image 20250623191852.png|500]]

Using **Greedy Best-First Search** with the _Manhattan Distance_ as the heuristic, the algorithm is biased to move **right**, since those nodes are heuristically closer to the goal. However, that path ends up being **longer** in total cost than the **left** path, which was initially less appealing. Greedy chose *â€œwhat looks best nowâ€*, not *â€œwhat will cost least overall*.
## The cost of taking a step
To solve this issue, **(A*)** algorithm bring to the table the $g(n)$ function, it tracks the cost of taking that step. This addition is crucial: it helps **(A*)** consider not only how **close** a node appears to the goal, but also **how expensive it was to get there** by adding it to the $h(n)$:
$$f(n) = g(n) + h(n)$$

Letâ€™s see how this changes the behavior:
![[Pasted image 20250623193154.png]]

Initially, the algorithm follows the **red path** (going right), since it seems cheaper. But as it progresses, the **combined cost** $f(n)$ becomes higher than the nodes on the **left**. At that point, **(A*)** smartly **redirects its search** and begins exploring the **green path**, which leads to the actual **shortest path**.
## Core Idea:
- Initiate a **min priority queue** prioritized by $f(n)$.
- Start a **visited set**.
- Add initial **node** to the priority queue.
- Add initial **node** to the visited set.
- While priority queue is not empty:
    - Remove min **node** $N$ by $f(n)$ from the priority queue
    - if **node** $N$ is the target return.
    - Get all **adjacent nodes** $N'$ from node $N$.
    - Add them to the priority queue
    - Add them to the visited set.

## Pseudocode
```ts
const comparatorByFn = (a: Node, b: Node) => {
    return (h(a) + g(a)) < (h(b) + g(b))
}

function aStarSearch(node: Node, target: Node) {
    const priorityQueue = new minPriorityQueue(comparatorByFn);
    priorityQueue.add(node)

    const visited = new Set<Node>();
    visited.add(node);

    while( !priorityQueue.isEmpty() ) {
        const currentNode = priorityQueue.extractMin();
        if (currentNode == target) return;

        for (const neighbor of currentNode.adjacent()) {
            if (visited.has(heighbor)) continue;

            priorityQueue.insert(neighbor);
            visited.add(neighbor);
        }
    }
}
```
